Collaborative pill · Rhonatherm team

From scattered folders to a shared Rhonatherm project
Unique coding, automatic OneDrive structure and a common base to later exploit project data with AI

José Manuel Segundo Santiago · Thirty-second collaborative contribution to the hub on project management, collaborative work and structured company data

The problem

Today each Rhonatherm project can start under a different name, be stored locally or in the cloud according to individual criteria and accumulate documentation without a shared structure. That fragments knowledge, reduces visibility and makes the organization depend too much on each salesperson's or prescriber's memory.

The consequence is not only document disorder. It also becomes harder to supervise what projects exist, what status they are in and how that information can be reused by other areas.

The proposal does not start with a shiny AI layer. It starts with something more serious: creating a common project register and an information structure on which AI can later deliver real value.

What the tool proposes

José Manuel proposes a web application to register Rhonatherm projects with automatic project code, basic project data, responsible salesperson, status, amount and notes. When the project is created, the tool would also generate its folder structure in OneDrive or SharePoint.

The core of the system is a unique and readable project code so anyone in the company can identify branch, province, year, sequence and project name without ambiguity.

What it brings immediately

Shared visibility

Management, branches and the commercial team can consult the same project register without depending on personal folders.

Less duplication

A common sequence avoids repeated names, inconsistent criteria and unnecessary document hunting.

Real collaborative work

Documentation stops living on one person's machine and becomes accessible with role-based permissions.

Foundation for future exploitation

Once structured, project data can feed analysis, follow-up workflows and more useful automations.

Why AI fits here

The document is clear about it: AI will not be truly useful while project information remains scattered. By contrast, a centralized historical base with date, area, salesperson, client, amount and status opens the door to predictive analysis, internal assistants, document generation and useful alerts for sales teams and management.

So the proposal does not use AI as decoration. It prepares the one prerequisite AI really needs: proprietary, structured and shared data.

How it coexists with HubSpot

The idea does not compete with the current CRM. It complements it. HubSpot would continue to manage contacts, opportunities and the commercial relationship; Rhonatherm Projects would organize the identified project itself, its code and its associated technical documentation.

That makes it possible to add an organizational layer of value without forcing the sales network to change its habits all at once.

Roadmap proposed

The vision is well sequenced: start with a pilot tool to register and code projects, then integrate it with SharePoint and role-based permissions, and only later activate the AI layer and more advanced reporting integrations.

That order is correct because it clearly separates the current operational need from the longer-term ambition.
